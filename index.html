2015年爱奇艺Java工程师内推面试经验

发表于 2015-09-15   |   分类于 面试   |  

今天是2015年9月15号，去爱奇艺上海参加内推的面试，之前问HR是哪个部门，HR说等面试的时候具体安排。但实际上内推的时候就已经决定了面试的部门了，去了服务云的项目组，听着他们主管介绍感觉项目组如此高端，以我的技术水平恐怕难以胜任该职位。事实上面试下来的感觉也并不是很好，估计也是那种被刷的吧。三轮技术型面试，全部是干货面试，没有谈人生谈理想的环节。其实每次面试的感觉都是，这特么是招应届毕业生么，然后过了一阵发现，原来我还没有达到应届毕业生的水平。所以在我面完爱奇艺之后，也顿时有这样的感觉。这次面试的是上海的岗位，问了一下他们的技术人员，好像上海这边做的内容和北京那边的内容都一样，因为上海这边收购了PPS，所以对PPS的内部员工做了整合。面试结果还没有出来，趁着记忆还清楚，回忆一下面试的内容。

Round-1

这一轮应该是一个工作不久的工程师，从面试题就可以看出还有以前找工作的面试题的套路。这部分基本与找工作的常规题类似，可惜准备不够充分还是有很多题目都不太会。

  1. 实现Java中Double.parseDouble(String str)方法，这部分代码在JDK中没有实现代码，因为FloatingDecimal是sun公司的私有类，看不到具体的实现代码。但是面试的时候也没要做一些异常判断，就实现最简单的字符串，里面只有小数点和数字的。
  2. 接下来是问了Java中堆内存的组成部分，包含那几个部分，在GC的时候对象是如何变化的。还有一些运行参数-xms，-xmn，-xmx的意义等等。还问了一下Java的基础题，Object类有哪些方法等。

  3. 剩下的都是算法题，内容比较杂。

    • 判断链表是否有环，以及有环时如何找到交点；
    • 利用两个栈实现队列，在倒栈时同时有数据插入该怎么处理；
    • 有一堆重量不一的石子要放入5个框内，如何放置可以保证框内石子重量最近似；
    • 有m行n列的数据，每列选取一个数字，找出一组所有列的数字和mod 7最小。

  4. 然后还有一些计算机网络的知识，如TCP协议的wait_closed和time_closed的区别，arp协议在哪一层，网络的五层分别传输的都是什么内容等等。

第一轮时间比较久，可能是我的实习经历和他们的工作并不是很相关吧。至于其他的知识，面试官说在他们部门，像hadoop，Spring这些都是默认会使用的，所有不需要问了。总得来说一面的问题都是面试常见的题目，还是技术太渣，准备不够，答的一般吧。

Round-2

第二轮面试仿佛应该是小部门的主管吧，年龄比较深沉，这里问的问题会比较偏开发一些，主要是Java的基础知识，可能我对EE不是很熟，问的都是SE的内容。

  1. 你对SE比较熟还是EE比较熟啊；
  2. ArrayList和LinkedList的区别；
  3. HashMap和Hashtable和CocurrentHashMap的之间的用法；
  4. TCP的三次握手和四次挥手；
  5. 写一个快速排序；
  6. SpringMVC的大概转发机制；
  7. 对hadoop有哪些了解；
  8. 未来的职业发展规划等等。

有印象的就是这些内容，可能还有一些不太记得了。和二面的面试官聊了一下爱奇艺的工作氛围，他之前在SAP做，感觉爱奇艺会比外企要更好一些，当然上下班打卡显得不是很open，但是觉得还OK，加班也有补贴，听着还不错啊，至少在互联网还不错啊。

Round-3

第三轮是最惨烈的一轮面试，这个是大部门的leader，我以为可以谈谈人生聊聊理想的层次了。还是问了我一个题目。问了很多技术层面的，但是没聊Java本身。

  • 计算机网络五层分别做了什么；
  • 操作系统进程间如何通信；
  • 实习中做了些什么，画一下技术架构之类的；
  • 一道大数据问题，1G的内存如何处理4G文件的数据，将数据排序。

第三轮最后一个问题的时候，想了很久，原来对这种大数据的题目没有太放在心上，结果这次吃了亏。可能这次的面试最大的问题就是在第三轮这里吧。

总结

整体感觉下来，可能爱奇艺的这个服务云的部门是偏基础架构的，问的问题都比较深入，对于应用层面的内容并不是很多。最后问了我是想从事业务级的开发还是技术型的开发，这个也是我现在所纠结的问题，希望尽快能有个idea吧。

Java设计模式学习（一） - 单例模式

发表于 2015-09-14   |   分类于 设计模式   |  

单例模式是设计模式中最简单的一个模式，但是面试笔试里经常会问这个知识点。本文主要针对于面试中常问的设计模式，也遵循一跳路的方案来理解什么是单例模式。

懒汉式单例1.0 —— 一般写法

单例模式最简单的一个方案就是懒汉式单例，这个是最容易想到的。但是其实对博主来说，懒汉饿汉这个名字很难分的清，我们姑且从最简单的开始分析吧。

这种单例最容易实现，只有在需要得到实例对象的时候才初始化对象，所以称之为”懒”，这个单例的问题显然就是对多线程的脆弱性。当遇到多线程访问时可能会破坏单例。

懒汉式单例2.0 —— 同步方法

很显然1.0版本对多线程不支持，如果thread1执行到底10行之后，thread2也执行到了第10行，这时候thread1会初始化对象，上下文切换之后，if(instance == null)对thread2并没有作用，也会新建对象，破坏单例。这时候出现了2.0版本。

2.0版本仅仅加入了synchronized关键字做了同步，这样对多线程就友好了，但同时效率不高，因为大部分时候执行的内容都是instance != null的。只有在第一次访问时才会新建对象，其余时候都是利用已经有的对象了，所以又有了下面这双重加锁的方式。

双重检验锁模式（double checked locking pattern），是一种使用同步块加锁的方法。程序员称其为双重检查锁，因为会有两次检查instance == null，一次是在同步块外，一次是在同步块内。为什么在同步块内还要再检验一次？因为可能会有多个线程一起进入同步块外的 if，如果在同步块内不进行二次检验的话就会生成多个实例了。

使用 volatile 的主要原因是其一个特性：禁止指令重排序优化。也就是说，在 volatile 变量的赋值操作后面会有一个内存屏障（生成的汇编代码上），读操作不会被重排序到内存屏障之前。从「happens-before」的角度理解的话，就是对于一个 volatile 变量的写操作都先行发生于后面对这个变量的读操作（这里的“后面”是时间上的先后顺序）。

饿汉式单例

这种方法非常简单，因为单例的实例被声明成 static 和 final 变量了，在第一次加载类到内存中时就会初始化，所以创建实例本身是线程安全的。这种写法的缺点是它不是一种懒加载模式（lazy initialization），单例会在加载类后一开始就被初始化，即使客户端没有调用 getInstance()方法。饿汉式的创建方式在一些场景中将无法使用：譬如 Singleton 实例的创建是依赖参数或者配置文件的，在 getInstance() 之前必须调用某个方法设置参数给它，那样这种单例写法就无法使用了。

静态内部类

这种写法仍然使用JVM本身机制保证了线程安全问题；由于静态内部类是私有的，除了getInstance()方法 之外没有办法访问它，因此它是懒汉式的；同时读取实例的时候不会进行同步，没有性能缺陷；也不依赖 JDK 版本。

防序列化机制

破坏单例模式的除了多线程之外，还有就是反序列化-序列化，当一个对象被序列化时也会破坏单例。这时候可以在单例类中增加readResolve()方法的方式解决。

readResolve()方法是用来替换从流中读取的对象的，它在readObject(ObjectInputStream)方法之后被调用，readObject()方法即从流中读取对象的方法。这样就可以避免使用序列化方式破坏单例。

最牛X的单例

以上的单例还是在运用各种技巧来实现，最后一种简直是利用规则来实现。这种代码也及其简洁，只需要三行就可以实现，但是可惜的是在面试中并不适用，因为很多面试官可能也并不了解这个特性，那就是枚举类。Java的枚举类是天生单例的，并且能够对多线程免疫，对序列化免疫，简直是神器。

这个时候是不是大家都想着无招胜有招呢。

总结

一般来说，单例模式有五种写法：懒汉、饿汉、双重检验锁、静态内部类、枚举。上述所说都是线程安全的实现，文章开头给出的第一种方法不算正确的写法。

一般情况下直接使用饿汉式就好了，但一般面试的话，面试官可能想要看到的是双重加锁和静态内部类的写法，枚举类装装X可以，面试还是少用吧。

全排列

字符串全排列问题，如[abc]三个元素的数组，会有[abc,acb,bac,bca,cab,cba]六种排列方式，全排列的实现方式很多，这里选取一个简单的递归解决方法，代码简单，比较容易理解。

递归的原理通过交换的方式实现全排列，将第i个字母与第一个字母交换后，对其余的子数组递归全排列。

组合算法

组合算法是取出一个数组的所有子数组的组合，这个在搜索引擎中经常使用，切词再拼词。基本方式就是[abc]的组合方式有[a,b,c,ab,bc,ac,abc]六种组合方式，总计2^n-1种组合方式。这里组合的方式采用位运算的操作，比递归要简单。

1<<buf.length相当于2^buf.length，表示共有这么多种组合情况，其二进制上的1表示每种情况取哪些字母。i & (1<<j)表示实际哪一位为1，取哪些位置的组合。

2015年阿里面试记录

发表于 2015-09-06   |   分类于 面试   |  

我参加2015年阿里巴巴校招的内推面试拿到Java研发工程师的offer，运气比较好的是恰好发offer的时候还没有通知缩招，所以拿到offer大部分靠运气吧。在这里主要是记录一下内推阿里面试的一些问题，据说明年阿里的校招会更加的凶残，可能面试会更加严格。

今年阿里的面试比较多，基本上都是4-6轮，我恰好是算多的，三轮部门技术面+一轮HR面+两轮交叉面试。因为内推是在暑假，正在公司实习，说实话没有怎么准备，基本上就凭着以前看的一些东西加上运气的成分一路过去。我从研一开始参加阿里的招聘，感觉上来说还是很不错的，面试官人都很nice，感觉就像聊天一样。之前屡战屡败，所以屡败屡战，终于守得云开见月明，所以才会写这篇博客。来捋一捋到底阿里的面试问了些啥，为什么老是过不了。

第一轮技术电面

第一轮面试早上10点钟，正在写着代码，突然收到阿里的电话，心里感觉没啥底啊，说实话一点都没准备。悲剧的事之前投简历的时候用的简历还是一年前的简历，上面的项目都还是很久远之前的，非常囧。

  • 自我介绍
  • 指针和引用之前有啥区别么
  • Java的基础知识
  • Spring和mybatis遇到问题怎么解决
  • JVM内存模型
  • 数据库的乐观锁与悲观锁
  • 红黑树和二叉平衡树

基本上阿里的面试都是自我介绍，自我介绍一般就是擅长的一直说，不擅长的干脆不要说。自我介绍应该是挖坑给面试官跳的过程，但是一般都是我自己挖坑给自己跳了。自我介绍吧嗒吧嗒说了一通，好像他没听进去。我之前的项目都是C语言的项目，ARM嵌入式啥的，但是我已经很久不写C了都忘得差不多，无奈这个面试官仿佛对C很有认识啊，一直问我。所以上面第二个问题就是扯C与Java之间有什么区别。Java的基础知识比较细，什么是多态，这个比较基础；然后就是Java的集合框架，基本上是面试绕不开的，看看JDK的源码了解一下；Java的多线程技术，有没有用过线程池，为什么用线程池，什么时候不适合用线程池。。。。

然后就是Java的框架，Spring和Mybatis基本上也就是用，遇到问题也就是在写的代码找错误，不太会调试源码，但是有一些错误还是和他交流了一下，比如Spring在用maven打包的时候在加载xml文件的时候会报找不到命名空间的异常，这种怎么解决的等等。

JVM内存模型这部分没有问的太深入，没有涉及到具体调优的方式和参数，基本就是以下一些问题：

  • 类的加载器加载方式，双亲委派机制，如果要加载远程URL的class怎么做;
  • JVM的内存模型，堆分为几个部分，为何会有s1和s2;;
  • 哪些GC方式，什么情况会触发full GC；
  • 你知道哪些OOM，就是堆，栈，DirectBuffer等；
  • 了解JVM对你的编程有什么帮助；

数据库问了一些索引相关的和锁，但是只问了乐观锁和悲观锁的概念方面，什么是事务解释一下ACID，索引的数据结构，B+树，B+树的数据是存放在哪里的等等。还有用过什么数据库调优的方式，慢查询怎么看，用过哪些mysql的命令，分析sql的查询计划会得到哪些参数，分别是做什么的，这个不需要全部，可以参考explain查询计划。

最后就是数据结构，问我比较了解什么数据结构，问了红黑树的性质，五条性质，什么是平衡树，为什么要有平衡树，他有什么好处等等。

一面主要是一些基础知识，阿里注重基础、潜力和兴趣，遇到不会的可以直接说不会，然后说说自己有什么想法。比如不知道数据库的乐观锁和悲观锁的实现机理，但是我知道Java中怎么实现的，所以就扯了一下。一面基本上都还OK，运气好碰到个nice的面试官都差不多的。

第二轮技术电面

第二轮面试应该是稍微牛一些的面试官，这个聊的Java内容比较多。才开始的时候就是说目前的实习，我在一家互联网公司实习，做的工作和该部门的比较类似，所以初始印象还不错。后面聊了一些技术问题，主要是介绍之前的项目，然后问了几个问题，有没有可以优化的地方或者考虑一下大数据高并发的场景。比如要实现一个用户登录的场景，该怎么实现，如何保证安全性；到第二步如果数据量非常大，比如过千万注册用户过亿注册用户该怎么办等等。之后问了我经常上哪些技术网站，用过哪些开源的库，哪里不错，有没有关心过源代码怎么实现的。基本上的问题就是为什么要用，能不能不用，用了有什么好处。

之后同样问了JVM的相关问题，可见阿里对JVM有多钟爱。问了一下一个类如果只有一个int变量，这个类占多大得空间，这个当时答的是空对象大小(8byte)+int大小(4byte)，也没说对不对。问了Hashmap的实现原理，当hashmap冲突很大时怎么办，什么时候rehash以及如何rehash等。

二面问的会深入一些，主要是项目的东西聊了比较多，看个人的状况了。项目没什么东西可能就会去深入问Java的一些基本知识吧。

第三轮技术电面

第三轮技术电面应该是部门主管面，在23:00打电话突然要面试，可见阿里加班之盛况。由于比较晚而且我在租的房子里没怎么听清，就随便聊了一下。问了一下比较关注的技术，和未来的想法。这部分感觉不太像技术型面试，聊了一下自己目前的技术状况，还有想做的事情和未来的计划。因为之前读过很多阿里的人写的有关Java开发的书，所以对阿里在Java方面的造诣很佩服，这个直说就OK，没有问太深入细节的技术问题。

HR面试

HR面试等了许久，都以为要挂了，突然HR晚上打电话面试。和HR聊了很多，因为实习一直在业务部门做技术，所以聊了一些业务和技术的问题。HR的问题是：

  • 目前的状况和未来的发展，是技术还是other;
  • 之后在公司可能会经常遇到业务的琐事大于技术方面的事，怎么看待；
  • 做了实习之后对自己的技术方面有什么帮助；

基本上HR面试就是聊聊人生，总之就是坦诚一些吧，有什么说什么。最后问了一下HR面究竟会不会刷人，给的答案也比较模糊。说HR也是有刷人的资格，还是综合考虑你之前的面试情况等等。

第一轮交叉面试

本来感觉HR面完已经没啥问题，可是突然在一个礼拜之后接到一个电话说还要面试，但是这一个礼拜压根也就没做这方面的准备啊。交叉面试就是不同部门的人来面试，时间不长，就问了几个问题：

  • 为什么要来阿里，未来的技术规划（我当时想说我选周杰伦。。）；
  • 在学习Java的过程中读过什么书；
  • 对什么数据结构比较熟悉，链表有什么操作；
  • 说说快速排序的思想，快速排序不适合于什么场景；
  • Java的Object类有哪些方法；
  • 什么时候需要重写hashcode和equals方法，为何要重写。

这部分都是比较基础的知识，但是说实话没想到还要交叉面试，而且这个面试官态度也比较高冷，最后也没问我还有什么问题就直接挂电话了。

第二轮交叉面试

几天之后又接到一轮面试，心想这应该是最后一轮了吧。这个面试官态度和蔼，聊了一下开源技术之类的东东，说什么用Spring做些什么，IOC和AOP怎么实现的啊，声名式事务的源码看过么。然后问了点数据结构知识，平时上什么技术网站。就这样很愉快的结束了最后一轮面试，坐等消息，三天之后状态变为待同意拿offer。

总结

经过这么多轮的折腾终于收获offer，但是在签offer之后的几天就收到了阿里的缩招通知。在写这篇博客的时候还没有收到HR的通知，不知道之后会有什么波折。唯一运气不错的就是没有通过阿里之前的实习生面试，如果通过了可能也就拿不到校招的offer了。最后会不会入职阿里不好说，还有很多需要学习的东西。加油！！
